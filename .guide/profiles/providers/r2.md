# r2.md

R2 provider profile for adapters that use Cloudflare R2 as an object storage provider.

## Scope

- object storage constraints specific to R2 usage through adapters
- object-key, metadata, and access-pattern considerations
- storage, deletion, and audit implications tied to R2 adoption

## Applies When

- An adapter stores, reads, deletes, or signs access to objects in R2.

## Baseline

- Keep provider-agnostic adapter responsibilities in `../../principles/roles/adapter.md`.
- Keep storage and deletion semantics in `../../principles/domains/data-governance.md`.
- Use this profile only for R2-specific concrete constraints.

## Provider Constraints

- Treat bucket layout, object-key layout, and metadata layout as explicit design decisions.
- Do not let bucket names or object keys implicitly redefine tenant boundaries.
- Keep tenant containment explicit in object-key design, metadata, and lookup paths.
- Keep object metadata limited to fields safe for persistence and retrieval.
- Do not store secrets, raw credentials, or authorization truth in object metadata.

## Access Constraints

- Keep direct object access explicit rather than exposing raw storage internals upstream.
- Use short-lived, scoped access delegation when signed access is required.
- Do not treat possession of an object URL as sufficient authorization truth.
- Re-check tenant and ownership constraints before issuing signed access or destructive operations.

## Deletion and Retention Constraints

- Keep delete semantics explicit for object bodies, metadata, versions, derived assets, and backups if present.
- Do not assume object overwrite, lifecycle policy, or eventual deletion satisfies product deletion semantics on its own.
- Keep retention and restore behavior explicit for every object class stored in R2.

## Observability Constraints

- Record object operations with request correlation, operation identity, and tenant context.
- Redact object names or paths when they contain user-derived or sensitive identifiers.
- Keep access logs and audit records distinct when object operations have irreversible or external-effect semantics.

## Repo-local Requirements

- Each repository or shared concrete spec that adopts this profile must define:
  - bucket layout
  - object-key patterns
  - metadata schema
  - signed-access policy
  - deletion and retention behavior by object class

## Non-goals

- runtime-platform constraints owned by `profiles/runtime/`
- identity-provider constraints
- repository-local object schemas or operation catalogs
